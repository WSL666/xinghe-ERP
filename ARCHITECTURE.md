# 高并发架构演进方案

> 本文档描述 Product Pipeline 从单体应用向高并发、可扩展架构的演进路线。
> 核心原则：**先瓶颈在哪补哪，单体扛不住再拆**。不盲目上微服务。

---

## 一、现状

- FastAPI 单体 + Redis 队列 + 单 worker 实例
- 采集与 AI 解耦：采集免费无限落库，AI 按需付费走队列
- systemd 部署，Caddy 反向代理
- 数据量：百级链接，单用户测试阶段

---

## 二、五层架构

### 第一层：采集层（IO 密集，瓶颈在网络）

| 要点 | 说明 |
|------|------|
| 采集异步落库 | FastAPI + httpx 异步并发，直接写库 `status='collected'`，不入队、不阻塞 |
| 采集与 AI 彻底解耦 | 采集只管写库，AI 由开关控制独立触发 |
| 批量采集接口 | 上万条时支持一次收一批（非逐条 POST），减少请求次数 |
| 限流 | 每用户/全局采集并发上限，Redis 令牌桶控制，防单用户刷爆 |

### 第二层：队列层（核心枢纽）

| 阶段 | 方案 |
|------|------|
| 当前（万级） | Redis List/Stream，够用 |
| 十万+ | 迁移 Kafka / RabbitMQ，持久化 + 分区 + 多消费者 |

**队列分类型设计（关键）：**

```
queue:title    ← AI标题（快，~1秒）
queue:image    ← AI生图（慢，~10秒+）
queue:vision   ← 视觉解析（中等）
```

- 不同类型分开队列，慢任务（生图）不会堵死快任务（标题）
- 每个队列独立 worker 池，各自扩缩容
- Redis 队列重启可能丢消息，量大了必须换 Kafka

### 第三层：AI Worker 层（计算密集，瓶颈在外部 API）

| 要点 | 说明 |
|------|------|
| 水平扩展 | worker 多实例，systemd 多起或 Docker Compose `scale=N` |
| 按队列分组 | `worker-title`、`worker-image`、`worker-vision` 独立扩缩容 |
| 并发数 = API 额度 | 不是越多越好，生图限 5 并发开 10 worker 没用反而被 429 |
| 熔断 + 重试 | 模型 API 挂了/限流，worker 熔断暂停消费，circuit breaker 模式 |
| 超时控制 | 每步超时，卡住标 error，不无限占用 worker |

### 第四层：数据库（最容易先扛不住）

| 优先级 | 措施 |
|--------|------|
| 必做 | 组合索引：`user_id` + `status` + `created_at`，列表查询必走索引 |
| 万级+ | 读写分离：主库写，从库读（列表、统计） |
| 冷热分离 | 已完成老数据归档到独立表/库，主表只留活跃数据 |
| 大字段 | AI 结果 JSON 拆表或压缩存储 |
| 百万级 | 分表（按用户或时间） |

### 第五层：API 服务层（多人并发）

| 要点 | 说明 |
|------|------|
| 多 worker | `uvicorn --workers 4` 或 gunicorn + uvicorn worker |
| 负载均衡 | Caddy/Nginx 前置 |
| 限流 | API 层按用户限流（每秒 N 次），防恶意刷接口 |
| 缓存 | 热点数据（用户信息、余额、配置）放 Redis，别每次查库 |
| 实时推送 | WebSocket / SSE 推进度，**禁止前端轮询查状态**（万人轮询会击穿） |

## 四、未来功能接入方案

### 单独生图 / 生视频

- 新增独立队列 `queue:image_standalone`、`queue:video`
- 新增独立 worker 池
- 前端新入口，后端复用 `llm/` + `tools/` 层，只加新 task 脚本
- 视频特别慢，设计为**异步任务 + 回调通知**，请求不能一直挂着

### 智能体聊天（实时交互）

- 与批量任务**完全不同**，走 **WebSocket / SSE**，不走队列
- 独立服务或路由组，LLM 流式输出
- 对话历史：Redis（短期）+ 数据库（长期）
- 独立资源池，不与采集队列混用

### Pod / 播客生成

- 长任务（几十分钟），走队列，异步生成
- 必须设计**断点续传 + 进度查询**
- 完成后通知用户

### 单独 LLM / 多模态模型扩展

- 在 `llm/` 层加新 client（按 API 协议分：openai_client / claude_client / gemini_client）
- 在 `tools/` 层加新 task 脚本调用对应 client
- `.env` 配置模型参数，factory 按需实例化

## 五、扩容优先级（按瓶颈出现顺序）

| 顺序 | 措施 | 触发条件 |
|------|------|----------|
| 1 | 加 worker 实例 | AI 处理速度跟不上采集速度 |
| 2 | 数据库索引 + 读写分离 | 列表查询变慢、万人查状态卡顿 |
| 3 | Redis 缓存热点数据 | 余额/配置频繁查库 |
| 4 | 队列分类型 | 生图任务堵死标题任务 |
| 5 | API 多 worker + 限流 | 并发用户增多、接口响应变慢 |
| 6 | 前端改 WebSocket/SSE | 前端轮询击穿服务 |
| 7 | 拆微服务 | 某模块资源需求差异极大（如 AI 要 GPU 机器） |

> **原则**：7 之前都不需要拆微服务。单体 + 模块化目录（llm/tools/store/platforms）+ 水平扩展 worker 足够撑到万级链接 + 几百并发用户。

## 六、现阶段基础工作（不打架，只打地基）

以下改动不影响现有功能，但为高并发做好准备：

- [ ] 数据库索引检查（user_id + status + created_at 组合索引）
- [ ] worker 配置成可多实例启动（systemd 模板实例化 `@` 语法）
- [ ] 队列按类型分 key（`queue:title` / `queue:image` / `queue:vision`）
- [ ] 每步 AI 调用加超时 + 重试 + 熔断
- [ ] 前端改 WebSocket/SSE 推进度（替代轮询）
- [ ] 监控：worker 队列堆积告警、模型 API 错误率告警
- [ ] 余额/配置查询加 Redis 缓存（TTL 60s）

## 七、何时拆微服务

只有满足以下**至少一条**才考虑拆：

- AI 服务需要部署在 GPU 机器，采集在普通机器（资源需求差异大）
- 多团队并行开发不同模块（组织规模驱动）
- 单服务 QPS 上千、单库数据上千万（性能驱动）
- 某模块需要独立发布周期（业务驱动）

> 当前阶段：**不需要拆**。保持单体 + 分层目录，做好 worker 水平扩展和数据库优化即可。
